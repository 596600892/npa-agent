# NPA Agent 产品需求与系统设计 V0.1

> 状态：草案
> 阶段：产品需求与系统设计
> 第一主场景：个贷/消费贷不良资产包
> 核心原则：用户傻瓜式操作，系统专业化判断

## 1. 产品定位

NPA Agent 是面向 AMC、个贷/消费贷资产包投资人、律所、电话调解与催收团队的本地优先不良资产 Agent 工作台。

它不是单一聊天机器人，也不是简单 Excel 分析工具，而是一个围绕不良资产全流程的智能工作系统：

```text
资产包数据进入
-> 字段识别与数据标准化
-> 个贷/消费贷画像分析
-> 数据完整度与可信度判断
-> 法院/管辖/处置模式分析
-> 电话调解策略与合规话术
-> 三情景报价建议
-> 报告输出与项目记录
-> 公司经验沉淀与后续自我进化
```

第一版聚焦个贷和消费贷资产包，不优先覆盖房抵、企业贷、破产重整和大型复杂抵押物处置。

## 2. 要解决的问题

### 2.1 投资决策问题

AMC 和投资人拿到资产包后，通常要快速判断：

- 这个包值不值得继续看。
- 当前数据够不够做初筛。
- 资产包金额结构是否适合公司。
- 债务人画像是否利于处置。
- 手机号、身份证、地址等关键字段完整度如何。
- 管辖法院是否集中，后续诉讼成本是否可控。
- 更适合电话调解、批量诉讼、分包清收还是低价清收。
- 建议收购价格区间大概在哪里。

### 2.2 处置执行问题

律所、调解和催收团队买后更关心：

- 哪些户先处理。
- 哪些户适合电话调解。
- 哪些户金额高、资料完整，适合重点攻坚。
- 哪些户缺少手机号或地址，触达难度高。
- 话术怎么写才合规、理性、可执行。
- 什么时候从调解转诉讼或分包。
- 处置过程如何沉淀为公司经验。

### 2.3 数据不足问题

现实中用户经常只能提供有限数据，例如：

```text
编号 + 本金
姓名 + 本金
姓名 + 身份证 + 本金
姓名 + 手机号 + 本金
姓名 + 身份证 + 手机号 + 地址 + 本金 + 利息
```

系统不能因为字段少就停止工作。NPA Agent 必须在有限数据下给出当前最优分析，并明确告诉用户：

- 当前能判断什么。
- 当前不能判断什么。
- 哪些结论可信度高。
- 哪些结论只是推断。
- 补哪些字段最有价值。
- 补充后分析完整度会如何提升。

## 3. 产品原则

### 3.1 用户傻瓜式操作

用户只需要：

- 说出目标。
- 上传资料。
- 根据提示做最少确认。

系统负责：

- 判断用户意图。
- 检查缺失资料。
- 自动字段映射。
- 弹出必要配置。
- 选择分析路径。
- 调用对应 skill。
- 给出结果和下一步动作。

### 3.2 最少打扰

如果已有数据足够进行初筛，系统先分析，不要求用户补齐所有资料。

只有当缺少关键字段导致无法继续时才打断用户。例如本金缺失时，系统必须要求确认字段或重新上传。

其他可选字段只提示影响：

```text
缺少手机号：无法准确评估电话调解触达率。
缺少身份证：无法准确分析年龄、性别、户籍地。
缺少管辖法院：无法判断诉讼集中度。
```

### 3.3 数据有出处

所有关键结论必须标注来源：

- Excel 上传数据。
- 合同或文书解析。
- 用户配置。
- 公司历史数据。
- 外部公开数据。
- 模型推断。
- 缺失假设。

报告不能只给结论，必须告诉用户结论从哪里来。

### 3.4 默认安全

系统默认采用本地优先策略：

- 身份证、手机号、地址、姓名默认脱敏展示。
- 身份证解析、手机号归属、基础画像默认本地处理。
- 云端模型调用默认使用脱敏内容。
- 原文云端分析必须用户确认。
- 所有 skill 权限可见。
- 所有分析过程留痕。
- 模型配置必须傻瓜化，用户只输入 API Key 或选择本地模型，系统自动判断 provider、可用模型与安全模式。

### 3.5 越用越懂公司

系统应逐步沉淀：

- 公司常看的资产类型。
- 常用报告格式。
- 地区偏好。
- 法院经验。
- 历史处置结果。
- 调解成功经验。
- 报价修正经验。

但敏感明文数据不默认长期保存。

## 4. 第一版目标用户

### 4.1 AMC 或投资人

关注点：

- 能不能买。
- 多少钱买。
- 风险在哪里。
- 需要补哪些尽调。
- 适合什么处置模式。

主要输出：

- 资产包初筛报告。
- 数据完整度报告。
- 三情景报价建议。
- 下一步尽调清单。

### 4.2 律所或调解团队

关注点：

- 先联系谁。
- 怎么分层。
- 电话调解怎么说。
- 哪些户转诉讼。
- 法院和管辖是否集中。

主要输出：

- 电话调解策略。
- 合规话术。
- 处置优先级。
- 重点户清单。

### 4.3 催收/清收团队

关注点：

- 联系方式完整度。
- 金额分层。
- 区域分布。
- 分包或外包价值。
- 低成本触达路径。

主要输出：

- 批量触达建议。
- 分包清收建议。
- 低价清收/放弃建议。

## 6. 用户体验设计

### 6.1 第一屏

第一屏为工作台，不是营销页。

核心区域：

- 对话输入框。
- 常用任务按钮。
- 最近项目。
- 上传入口。
- 安全模式提示。

常用任务：

```text
分析资产包
查看新机会
生成处置策略
查看历史项目
生成电话调解方案
```

### 6.2 典型流程

用户输入：

```text
帮我看看这个个贷资产包能不能买。
```

系统流程：

```text
1. 检查是否已有项目或上传文件。
2. 没有文件则弹出上传 Excel。
3. 上传后自动识别字段。
4. 如果本金字段明确，直接进入分析。
5. 如果本金字段不明确，弹字段确认卡。
6. 默认确认资产类型为个贷/消费贷，可允许用户修改。
7. 执行基础画像、金额、触达、处置模式分析。
8. 生成报告。
9. 告诉用户缺哪些字段会更准。
10. 保存项目记录。
```

### 6.3 配置卡片

配置不应一次性展示复杂表单，而是在任务中按需弹出。

示例：

```text
我识别到这个资产包可能是个贷/消费贷。
如果不准确，请修改资产类型。

[个贷/消费贷] [房抵] [企业贷] [混合资产包]
```

字段确认示例：

```text
我不确定哪个字段是本金。
请确认下面哪个字段代表本金余额。

[未偿本金] [剩余本金] [债权本金] [其他]
```

缺失提示示例：

```text
当前可以完成 Level 1 初筛。
如果补充身份证和手机号，可以继续做人群画像和电话调解可行性分析。
```

## 7. 数据输入

### 7.1 MVP 必须支持

资产包 Excel 清单。

### 7.2 基础字段模板

```yaml
loan_account:
  debtor_name_or_id: string | required
  id_card: string | optional_sensitive
  phone: string | optional_sensitive
  address: string | optional_sensitive
  principal: number | required
  interest: number | optional
```

字段规则：

- 债务人名称没有时，可用编号替代。
- 身份证没有时，可用年龄、性别、地址等字段替代部分画像。
- 手机号没有时，不阻塞分析。
- 本金必须有。
- 利息可选。

### 7.3 推荐增强字段

```yaml
optional_fields:
  contract_no: string
  overdue_days: number
  loan_date: date
  maturity_date: date
  jurisdiction_court: string
  arbitration_body: string
  litigation_stage: string
  execution_court: string
  repayment_history: string
  remark: string
```

### 7.4 字段自动识别

字段别名示例：

```yaml
debtor_name_or_id:
  - 债务人名称
  - 客户名称
  - 姓名
  - 借款人
  - 客户编号
  - 借据编号

id_card:
  - 身份证
  - 身份证号
  - 证件号码
  - 证件号

phone:
  - 手机号
  - 手机号码
  - 联系电话
  - 电话

address:
  - 地址
  - 户籍地址
  - 居住地址
  - 通讯地址

principal:
  - 本金
  - 本金余额
  - 未偿本金
  - 剩余本金
  - 债权本金

interest:
  - 利息
  - 欠息
  - 利息余额
  - 罚息
```

低置信度字段必须让用户确认。

### 7.5 数据等级

```text
Level 1：本金/编号
能做金额规模、户均本金、粗略处置模式。

Level 2：身份证/手机号/地址
能做人群画像、触达率、区域集中度、电话调解可行性。

Level 3：合同/法院
能做管辖集中度、诉讼便利度、法院画像。

Level 4：公司历史处置数据
能做公司专属回收率、法院经验、报价校准。

Level 5：外部公开/商业数据
能做债务人风险、类案、执行压力、法院饱和度增强判断。
```

- 识别用户意图。

- 判断缺失数据。
- 弹出配置卡。
- 选择 skill。

## 10. 分析算法

### 10.1 数据完整度评分

建议初始权重：

```yaml
data_completeness:
  principal: 30
  debtor_name_or_id: 10
  id_card: 20
  phone: 20
  address: 10
  interest: 5
  court_or_contract: 5
```

本金缺失时，不给完整报告，只进入字段确认流程。

### 10.2 金额结构

指标：

- 总本金。
- 总利息。
- 本息合计。
- 户数。
- 户均本金。
- 中位数本金。
- 高金额户占比。
- 小额户占比。
- 金额分布区间。

默认金额分层：

```text
0 到 5000
5000 到 10000
10000 到 30000
30000 到 100000
100000 以上
```

### 10.3 人群画像

身份证可解析时：

- 性别。
- 年龄。
- 年龄段。
- 户籍地。

年龄段：

```text
18-25
26-35
36-45
46-55
56-65
65+
未知
```

身份证不可用时：

- 使用年龄字段。
- 使用性别字段。
- 使用地址字段。
- 可信度降为中或低。

### 10.4 触达能力

指标：

- 手机号完整率。
- 地址完整率。
- 身份证完整率。
- 同一手机号重复率。
- 空号或异常格式比例。

手机号格式仅做基础校验，不代表可真实触达。

### 10.5 地区集中度

来源优先级：

```text
1. Excel 明确地区/地址字段。
2. 身份证前 6 位户籍地。
3. 合同/法院字段。
4. 用户历史数据。
```

报告必须说明地区来源。

### 10.6 处置模式判断

电话调解优先：

- 手机号完整率较高。
- 单户金额中低。
- 地区或法院分散。
- 诉讼成本不经济。
- 当前缺少合同/法院材料。

批量诉讼：

- 管辖法院集中。
- 合同或证据链完整。
- 单户金额足以覆盖诉讼成本。
- 公司历史显示该法院适合批量处理。

分包清收：

- 地区高度分散。
- 户数多。
- 单户金额偏小。
- 联系方式一般。

低价清收/放弃：

- 数据严重缺失。
- 本金金额低。
- 手机和地址缺失。
- 管辖不明。
- 补充数据成本高。

重点户攻坚：

- 高金额户。
- 信息完整。
- 地址或法院明确。
- 可能适合单独尽调。

### 10.7 三情景报价

MVP 不做复杂 IRR，先用规则区间。

输出：

```yaml
pricing_scenarios:
  conservative:
    recovery_rate: string
    cycle_months: string
    suggested_bid_rate: string
  base:
    recovery_rate: string
    cycle_months: string
    suggested_bid_rate: string
  optimistic:
    recovery_rate: string
    cycle_months: string
    suggested_bid_rate: string
```

报价必须提示：

```text
本报价基于当前数据和模型假设，不构成投资建议。
```

### 12.2 分层

建议分层：


禁止：

- 威胁、恐吓、侮辱。

- 冒充司法机关。
- 向无关第三人泄露债务信息。
- 使用不确定事实施压。
- 暗示违法后果。

推荐表达：

- 核验事实。
- 说明债权信息。
- 询问还款意愿。
- 提供协商空间。
- 说明可选方案。
- 保留后续依法处理权利。

```text
广东省深圳市南山区... -> 广东省深圳市南山区***
```

### 14.3 Skill 权限

每次 skill 调用必须记录：

- skill 名称。

- 是否访问敏感字段。
- 是否联网。
- 是否写入记忆。
- 输出报告版本。

### 14.4 云端调用

默认：

- 使用脱敏内容。
- 原文云端调用必须确认。
- 审计日志记录确认动作。

### 14.5 模型配置

模型配置遵循“用户简单输入，系统专业判断”的原则。

用户侧只需要：

- 输入 API Key。
- 或选择本地模型路径/本地模型服务地址。
- 或从系统推荐列表里选择一个模型。

系统侧负责：

- 自动识别 API Key 对应的 provider。
- 自动探测可用模型。
- 根据任务类型推荐模型。
- 根据安全模式决定是否允许发送原文。
- 根据文档长度、隐私等级和成本约束选择模型。
- 在用户不懂模型差异时提供默认推荐。

支持的配置形态：

```yaml
model_config:
  mode: local | redacted_cloud | original_cloud
  provider: auto | openai | deepseek | qwen | minimax | moonshot | zhipu | custom | local
  api_key: encrypted_string | null
  base_url: string | null
  model: auto | string
  allow_original_sensitive_data: false
```

默认模式：

```text
redacted_cloud
```

即默认允许脱敏后的内容调用云端模型，不默认发送原始身份证、手机号、地址、姓名和完整资产清单。

安全等级：

```text
本地模式：
敏感数据不出本机，适合最高安全要求。

脱敏云端模式：
默认推荐。敏感字段先脱敏，再调用云端模型。

原文云端模式：
高风险。只有用户明确确认后才能启用，并记录审计日志。
```

模型选择逻辑：

```text
字段识别、金额统计、身份证解析：
优先本地规则，不调用大模型。

报告总结、话术生成、合同条款解释：
可使用脱敏云端模型。

包含原始合同、完整债务人信息、完整资产清单：
默认本地或脱敏处理，原文云端必须确认。
```

配置交互示例：

```text
请输入你的模型 API Key。我会自动识别服务商和可用模型。
如果你不确定用哪个模型，保持“自动推荐”即可。
```

当用户启用原文云端模式时，必须提示：

```text
当前资料可能包含身份证、手机号、地址和债务人信息。
如果选择原文云端分析，原始敏感信息会发送到外部模型服务。
建议使用“脱敏云端分析”或“本地分析”。
```

凭证安全：

- API Key 只保存在本地配置或本地加密存储中。
- 报告、日志和错误信息不得明文展示完整 API Key。
- Skill 不得直接读取 API Key，必须通过模型网关调用。
- 每次模型调用记录 provider、model、安全模式、是否脱敏，但不记录完整请求敏感原文。

## 16. 语音对话方案

NPA Agent 需要支持语音，但语音不是唯一入口。语音设计遵循“双方案”：

```text
推荐方案：Bailongma 风格云端/流式语音，效果更好。
备选方案：浏览器或系统自带语音，配置简单但效果较弱。
```

### 16.1 方案 A：Bailongma 风格语音增强

Bailongma 当前语音能力包含：

- 本地 Whisper 语音服务进程管理。
- 云端 ASR WebSocket 代理。
- 阿里云、腾讯云、讯飞 ASR provider。
- 豆包、MiniMax、OpenAI、ElevenLabs、火山引擎 TTS provider。
- 流式 TTS。
- 语音输入面板。
- 说话状态、识别状态、AI 播报状态。
- TTS 播放中用户打断检测。

NPA Agent 推荐复用其设计思路，但不直接把不良资产数据交给语音 provider。语音层只处理用户说的话和系统播报内容，不默认读取或发送完整资产包原始数据。

适用场景：

- 老板或投资经理口头询问：“这个包能不能看？”
- 调解团队口头生成话术。
- 系统播报：“这个资产包当前数据完整度 62%，建议补充身份证和手机号。”
- 后续自动播报银登网新机会。

推荐配置：

```yaml
voice_enhanced:
  asr_provider: auto | aliyun | tencent | xunfei | local_whisper
  tts_provider: auto | doubao | minimax | openai | elevenlabs | volcano
  tts_voice: auto | string
  streaming: true
  barge_in: true
  sensitive_data_readout: masked_only
```

安全要求：

- 语音播报默认只读脱敏摘要，不播报完整身份证、手机号、地址。
- 语音识别文本进入普通任务编排前，要经过敏感词和意图识别。
- 语音 provider 的 API Key 与模型 API Key 分开保存。
- 语音 provider 不得接收完整资产包文件。
- 语音调用写入审计：provider、时间、是否播报敏感摘要、是否使用云端。

配置交互：

```text
你可以启用增强语音。只需要填写语音服务的 API Key，我会自动识别可用服务。
如果暂时不配置，也可以使用浏览器自带语音，效果会弱一些。
```

### 16.2 方案 B：自带语音备选

自带方案用于低门槛启动，不依赖额外语音服务。

可选实现：

- 浏览器 Web Speech API 做基础 ASR/TTS。
- 系统自带语音播报。
- 本地 Whisper 简化模式。

优点：

- 不需要额外 API Key。
- 更适合开源版开箱体验。
- 敏感数据更容易限制在本地。

缺点：

- 中文识别效果不稳定。
- 流式体验较弱。
- 声音自然度和可控性较差。
- 部分浏览器或系统兼容性不一致。

默认策略：

```text
未配置增强语音时，自动启用自带语音备选。
配置增强语音后，优先使用增强方案。
语音失败时回退到文本。
```

### 16.3 NPA 语音交互边界

允许：

- 语音发起任务。
- 语音询问报告摘要。
- 语音生成电话调解话术。
- 语音播报缺失字段和下一步建议。

默认不允许：

- 播报完整身份证。
- 播报完整手机号。
- 播报完整地址。
- 在公共环境下朗读完整资产包清单。
- 用语音直接确认高风险原文云端调用。

高风险操作必须回到界面确认：

- 原文云端模型分析。
- 写入长期记忆。
- 启用外部 skill。
- 导出包含敏感信息的报告。

## 19. 实现级补充规格

### 19.1 标准 Excel 模板

MVP 应内置标准模板下载，但不强迫用户必须按模板上传。模板既服务小白用户，也作为字段映射引擎和测试样例的基准 schema。

标准模板字段：

| 字段名 | 是否必填 | 类型 | 说明 | 缺失影响 |
|---|---:|---|---|---|
| 债务人名称 | 否 | 文本 | 债务人姓名；没有姓名时可用债务人编号替代 | 不能做人名脱敏展示，报告以编号展示 |
| 债务人编号 | 否 | 文本 | 内部编号、客户编号、借据编号均可 | 与债务人名称二选一；两者都缺时系统自动用行号生成编号 |
| 身份证号 | 否 | 文本 | 用于年龄、性别、户籍地分析 | 无法准确做身份证画像，使用年龄/性别/地址替代 |
| 手机号 | 否 | 文本 | 用于触达率、电话调解可行性分析 | 电话调解评分下降 |
| 地址 | 否 | 文本 | 户籍/居住/通讯地址均可 | 地区集中度可信度下降 |
| 本金 | 是 | 数值 | 未偿本金、本金余额、债权本金均可 | 无法完成资产包金额分析，必须确认或补充 |
| 利息 | 否 | 数值 | 欠息、罚息、利息余额均可 | 只按本金做分析，报价保守 |
| 逾期天数 | 否 | 数值 | 用于判断催收阶段和诉讼时效线索 | 无法判断逾期结构 |
| 合同编号 | 否 | 文本 | 用于关联合同/文书 | 无法做合同文件匹配 |
| 管辖法院 | 否 | 文本 | 合同约定或案件法院 | 无法判断诉讼集中度 |
| 备注 | 否 | 文本 | 其他线索 | 仅作为辅助文本 |

导入规则：

- 本金是唯一强制业务字段。
- 债务人名称和债务人编号二选一；都缺失时系统用 `ROW-行号` 作为临时编号。
- 数值字段支持中文金额、带逗号数字、人民币符号。
- 空白行跳过。
- 重复债务人不自动合并，先作为单户记录保留，并在报告中提示可能重复。

### 19.3 数据完整度评分

基础公式：

```yaml
data_completeness_score:
  principal: 30
  debtor_identifier: 10
  id_card: 20
  phone: 20
  address: 10
  interest: 5
  court_or_contract: 5
```

评分规则：

- 本金字段存在且 90% 以上行可解析：得 30 分。
- 债务人名称或编号覆盖率 90% 以上：得 10 分。
- 身份证覆盖率按比例计分，满分 20。
- 手机号覆盖率按比例计分，满分 20。
- 地址覆盖率按比例计分，满分 10。
- 利息覆盖率按比例计分，满分 5。
- 管辖法院、合同编号、执行法院任一关键字段覆盖率 50% 以上：得 5 分。

数据等级：

```text
0-35：Level 1，基础金额初筛。
36-65：Level 2，可做人群画像和电话调解初判。
66-80：Level 3，可做较完整处置策略。
81-90：Level 4，可结合历史数据校准。
91-100：Level 5，适合深度尽调和报价测算。
```

### 19.4 个贷处置模式默认阈值

第一版使用可解释规则，不使用黑箱模型。

电话调解优先：

```text
手机号覆盖率 >= 60%
且户均本金 <= 30000
且法院/管辖字段缺失或分散
```

批量诉讼优先：

```text
管辖法院覆盖率 >= 60%
且前 3 个法院覆盖 >= 70%
且户均本金 >= 30000
且合同编号覆盖率 >= 50%
```

分包清收优先：

```text
户数 >= 500
且地区分散
且户均本金 <= 20000
且手机号覆盖率在 30%-60%
```

低价清收/谨慎：

```text
数据完整度 < 35
或手机号覆盖率 < 20%
或本金字段异常率 > 20%
```

重点户攻坚：

```text
单户本金进入前 5%
且至少具备身份证/手机号/地址中的两个字段
```

这些阈值为默认初始规则，后续通过公司历史处置数据校准。

### 19.5 三情景报价默认规则

MVP 报价只做区间和假设，不做确定性投资建议。

```yaml
pricing_defaults:
  conservative:
    expected_recovery_rate: 0.03-0.06
    expected_cycle_months: 18-36
    suggested_bid_rate: 0.01-0.03
  base:
    expected_recovery_rate: 0.06-0.12
    expected_cycle_months: 12-24
    suggested_bid_rate: 0.03-0.06
  optimistic:
    expected_recovery_rate: 0.12-0.20
    expected_cycle_months: 6-18
    suggested_bid_rate: 0.06-0.10
```

修正因子：

- 手机号覆盖率高：提高电话调解回收假设。
- 年龄集中在 26-45：提高协商触达假设，但不得作为歧视性决策依据。
- 地区集中：降低处置成本假设。
- 管辖集中：提高批量诉讼可行性。
- 数据完整度低：降低报价区间。
- 公司历史同类回收率高：提高基准情景。

报告提示：

```text
报价为模型假设下的初筛区间，不构成投资建议。实际报价需结合合同、债权真实性、诉讼时效、历史回款和外部尽调结果复核。
```

### 19.6 报告样例骨架

报告首页摘要样例：

```text
推荐等级：B-
建议动作：可进入初步尽调，但暂不建议直接报价。

核心机会：
1. 本金规模适中，户均本金 18,600 元，适合电话调解优先。
2. 手机号覆盖率 72%，具备批量触达基础。
3. 债务人年龄主要集中在 26-45 岁，可作为协商触达辅助线索。

核心风险：
1. 未提供合同和管辖法院，无法判断诉讼集中度。
2. 身份证覆盖率仅 48%，区域画像可信度中等。
3. 未提供历史处置数据，报价只能使用行业规则区间。

下一步：
1. 补充身份证或年龄字段，提升人群画像准确度。
2. 补充合同编号和管辖法院，判断是否适合批量诉讼。
3. 上传公司历史同类资产包回款数据，校准报价区间。
```

数据来源样例：

```text
结论：手机号覆盖率 72%。
来源：Excel 手机号字段。
可信度：高。

结论：适合电话调解优先。
来源：手机号覆盖率、户均本金、法院字段缺失，规则模型推断。
可信度：中。

结论：诉讼集中度无法判断。
来源：缺少合同/法院字段。
可信度：低。
```

### 19.7 公司历史数据模板

系统应提供模板，但也支持用户上传非标准历史数据后自动识别。

| 字段名 | 是否必填 | 说明 |
|---|---:|---|
| 项目名称 | 否 | 历史资产包或项目名称 |
| 资产类型 | 否 | 个贷、消费贷、房抵、企业贷等 |
| 地区 | 否 | 主要处置地区 |
| 法院 | 否 | 主要管辖或执行法院 |
| 户数 | 否 | 历史项目户数 |
| 本金总额 | 否 | 历史项目本金 |
| 成交价 | 否 | 实际收购价 |
| 回款金额 | 否 | 实际回款 |
| 回款周期（月） | 否 | 从收购到主要回款周期 |
| 处置方式 | 否 | 电话调解、诉讼、执行、分包等 |
| 调解成功率 | 否 | 可选 |
| 诉讼成功率 | 否 | 可选 |
| 执行结果 | 否 | 可选 |
| 失败原因 | 否 | 例如失联、无财产、法院慢、证据不足 |
| 备注 | 否 | 人工经验 |

系统可从不完整历史数据中提取历史折扣率、历史回收率、地区经验、法院经验、金额段表现和处置方式效果。

```text
配置模型

```

## 21. 当前明确决策

- 第一版聚焦个贷/消费贷。
- MVP 必须支持 Excel 上传。
- 本金为唯一强制业务字段。
- 债务人名称可用编号替代。
- 身份证、手机号、地址可选，但会影响分析完整度。
- 电话调解第一版只做策略与话术。
- 报价采用规则区间 + 三情景。
- 报告先摘要，再详细分析，再来源说明。
- 默认本地优先、敏感数据脱敏。
- 模型配置由用户输入 Key 或选择本地模型，系统自动识别 provider 并推荐模型。
- 默认模型安全模式为脱敏云端；原文云端必须二次确认。
- 语音支持双方案：增强语音优先，自带语音保底。
- 增强语音借鉴 Bailongma 的云端 ASR、流式 TTS、本地 Whisper 和打断检测设计。
- 语音默认只播报脱敏摘要，不播报完整敏感字段。
- 第一版不自动安装 community skill。
- 标准 Excel 模板内置下载，但用户可上传任意格式。
- 第一版默认金额分层为 0-5000、5000-10000、10000-30000、30000-100000、100000 以上。
- 公司历史数据提供模板，同时支持非标准上传自动识别。
